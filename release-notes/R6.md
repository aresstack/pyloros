# Release 6 – Java 21 ACP Manager Agent

Release 6 ersetzt den nicht weiter verfolgten Release-5-Ansatz eines fest eingebauten LangChain-/Ollama-Providers im Pyloros-Core durch eine ACP-basierte Manager-Agent-Architektur.

Pyloros bleibt damit bewusst ein MCP Capability Gateway und Tool-Aggregator. Agentenlogik läuft außerhalb des Core als separater ACP-Agent und erhält den Pyloros-MCP-Aggregator als injizierten Toolserver.

## Highlights

- R5 LangChain-in-Core wurde verworfen und dokumentiert als superseded markiert.
- Der Core-Provider-Typ wurde auf die tatsächlich implementierten Provider reduziert: `NATIVE`, `MCP`, `ACP`, `UNKNOWN`.
- Neues Java-21-Modul `pyloros-manager-agent` als eigenständiger Gradle-`application`-Bootstrap.
- Minimaler ACP-Handshake für `session/new`, `session/prompt` und `session/update`.
- Pyloros injiziert seinen MCP-Aggregator über `mcpServers.pyloros` in die ACP-Session.
- `tools/list` und `tools/call` sind jetzt Tool-View-aware.
- Agent-Tool-Views werden gegen Selbstrekursion und Cross-ACP-Rekursion gehärtet.
- Nicht-öffentliche MCP-Views sind durch einen serverseitigen Shared-Secret-Token geschützt.
- End-to-End-Smoke-Test für den Minimalfluss Agent → MCP `tools/list` → MCP `tools/call` → Agent-Antwort.

## Architekturänderung

Vor R6 war Release 5 als eingebauter Natural-Language-/LangChain-Provider im Pyloros-Core gedacht. Dieser Weg wurde aufgegeben.

Ab R6 gilt:

```text
Client
  -> Pyloros MCP endpoint
    -> ACP virtual provider
      -> Java 21 ACP manager agent
        -> injected Pyloros MCP endpoint
          -> tools/list
          -> tools/call
```

Pyloros besitzt weiterhin:

- ProviderRegistry
- ToolCatalog
- ToolRouter
- MCP-Upstream-Aggregation
- Plugin-Provider
- ACP-Virtual-Provider
- Tool-View-Scoping
- Prefix-/Alias-/Collision-Schutz

Der Manager Agent konsumiert diese Capability-Oberfläche über MCP. Er dupliziert keine Providerverwaltung und keinen eigenen ToolCatalog im Core.

## ProviderType-Vereinfachung

Der Core-Enum `ProviderType` enthält nur noch:

```text
NATIVE,
MCP,
ACP,
UNKNOWN
```

Entfernt wurden:

```text
LANGCHAIN
COMPOSITE
REST
```

Diese Typen waren nicht produktiv implementiert und sollen nicht als Core-Konzepte reserviert bleiben. LangChain/Ollama, REST-Adapter oder Composite-Verhalten können später über Plugins, ACP-Agenten oder eigene Runtime-Erweiterungen entstehen.

## Neues Modul: `pyloros-manager-agent`

R6 führt ein neues, separates Java-21-Subprojekt ein:

```text
pyloros-manager-agent
```

Eigenschaften:

- Gradle-`application`-Modul
- eigener Startscript-/Distribution-Pfad
- keine Kopplung an `pyloros-server` als Runtime-Abhängigkeit
- stdout ist für line-delimited JSON-RPC reserviert
- Logging geht nach stderr
- minimaler ACP-Handshake statt vollständiger produktiver ACP-SDK-Runtime

Das Modul ist ein Bootstrap und eine technische Basis für spätere Agentenlogik. Es ist noch kein vollständiger Planner, Workflow-Kern oder LLM-Agent.

## Minimaler ACP-Handshake

Der Manager Agent unterstützt in R6 den minimalen Session-Flow:

```text
session/new
session/prompt
session/update
completion/error
```

Dabei werden Fehler strukturiert zurückgegeben, zum Beispiel bei fehlender Session, fehlendem Prompt oder fehlender injizierter MCP-Konfiguration.

Wichtig: Dieser Handshake bedient bewusst den vorhandenen Pyloros line-delimited JSON-RPC-stdio-Adapter. Er ist noch keine vollständige produktive ACP-SDK-Runtime.

## MCP-Injection in ACP-Sessions

Pyloros kann dem ACP-Agenten beim `session/new` einen MCP-Server injizieren:

```json
{
  "mcpServers": {
    "pyloros": {
      "url": "http://127.0.0.1:8080/pyloros?view=agent",
      "headers": {
        "Authorization": "Bearer ...",
        "X-Pyloros-Injected-View": "acp",
        "X-Pyloros-Injected-View-Token": "..."
      }
    }
  }
}
```

Der Manager Agent verwendet diesen Endpunkt für:

- `tools/list`
- `tools/call`

Der Smoke-Test ruft als sicheres Beispieltool `pyloros__ping` auf.

## Tool Views und MCP-Routing

R6 macht Tool-Zugriffe View-aware:

- `tools/list` verwendet eine konkrete `ToolView`.
- `tools/call` dispatcht aus dem Snapshot derselben `ToolView`.
- Per-View-Snapshots werden getrennt geführt.
- Leere Views löschen ihren alten Snapshot, damit keine veralteten Tools sichtbar bleiben.

Damit wird verhindert, dass ein vorheriger `tools/list`-Aufruf aus einer anderen View den späteren `tools/call`-Dispatch beeinflusst.

## Trust Boundary für nicht-öffentliche Views

Der öffentliche MCP-Endpunkt bleibt standardmäßig auf `public` beschränkt.

Ein nicht-öffentlicher `?view=...`-Parameter wird nur akzeptiert, wenn alle Bedingungen erfüllt sind:

1. Request kommt von Loopback.
2. Header `X-Pyloros-Injected-View: acp` ist gesetzt.
3. Header `X-Pyloros-Injected-View-Token` enthält den serverseitig konfigurierten oder generierten Token.
4. Der Tokenvergleich erfolgt constant-time über `MessageDigest.isEqual`.

Ohne gültigen Token fällt der Request auf `PUBLIC` zurück. Ein lokal authentifizierter Client kann die Agent-View also nicht allein durch `?view=agent` oder einen spoofbaren Marker-Header auswählen.

Der Token wird beim Start erzeugt oder über `MCP_INJECTED_VIEW_TOKEN` beziehungsweise `mcp.injected-view.token` konfiguriert und nur in die vom ACP-Provider injizierte `session/new`-MCP-Konfiguration geschrieben.

## Rekursionsschutz für ACP-Provider

R6 härtet `AgentToolViewValidator`.

Verhindert werden insbesondere:

- `agentToolView = public`
- Agent-View verweist auf die eigene Provider-ID
- Agent-View verweist auf eine andere ACP-Provider-ID
- ACP-Provider exponiert sich selbst in seiner Agent-View
- anderer ACP-Provider ist in der Manager-Agent-View sichtbar

Damit soll der Manager Agent weder sich selbst noch andere rekursionsgefährliche ACP-Agenten über MCP aufrufen können.

## Smoke- und Regressionstests

R6 ergänzt Tests für den neuen Minimalfluss und die neuen Sicherheitsgrenzen:

- Manager-Agent-Handshake ohne Pipe-Transport
- line-delimited JSON-RPC-Protokollserver
- injected-MCP-End-to-End-Smoke-Test
- `tools/list` + sicheres `tools/call`
- Header-Passthrough zum injizierten MCP-Endpunkt
- ToolRouter-View-Scoping

- ACP-Agent-View-Validator-Regeln
- Trust-Gate für nicht-öffentliche MCP-Views
- OAuth/Public-Endpoint-Kompatibilität mit Token-geschütztem Agent-View-Zugriff

## Nicht-Ziele von R6

R6 liefert bewusst noch nicht:

- vollständigen Workflow-/WorkGraph-Kern
- LLM-Integration
- LangChain-/Ollama-Core-Integration
- A2A-Agent-Delegation
- langfristige Persistenz oder Memory
- RAG
- Chat-History-UI
- produktionsreife ACP-SDK-Runtime
- generische Policy-Engine für alle Tool Views

## Upgrade-Hinweise

Für bestehende Konfigurationen wichtig:

- Core-Referenzen auf `ProviderType.LANGCHAIN`, `ProviderType.COMPOSITE` oder `ProviderType.REST` sind nicht mehr gültig.
- Externe oder virtuelle Provider sollten weiterhin per Prefix exponiert werden.
- Root-Tools ohne Prefix bleiben ein bewusstes Opt-in und sind nicht der Default für externe Provider.
- Nicht-öffentliche MCP-Views können nicht mehr allein per `?view=...` ausgewählt werden.
- Für lokale injected-agent Calls muss Pyloros den Shared-Secret-Token in `session/new` injizieren.

## Bekannte Einschränkungen

- Der Manager Agent wählt im Minimal-Smoke-Flow nur ein sicheres bekanntes Tool aus, bevorzugt `pyloros__ping`.
- Die Fallback-Erkennung für sichere Tools ist noch absichtlich minimal.
- Das globale Legacy-`ToolCatalog.snapshot()` existiert weiterhin; der sicherheitsrelevante Dispatch nutzt aber view-spezifische Snapshots.
- Die ACP-SDK-Dependencies sind gepinnt, aber der produktive ACP-SDK-Agent ist noch ein späterer Ausbau.

## Ergebnis

Release 6 etabliert die neue Agentenarchitektur:

```text
Pyloros = MCP Capability Gateway
Manager Agent = separater ACP-Agent
Toolzugriff = injizierter Pyloros-MCP-Aggregator
Sicherheit = Tool Views + Rekursionsschutz + Token-geschützte View-Auswahl
```

Damit ist die Grundlage gelegt, um spätere Manager-Agent-Funktionen außerhalb des Pyloros-Core weiter auszubauen, ohne die Core-Architektur mit konkreten LLM-, LangChain- oder Planner-Entscheidungen zu vermischen.
